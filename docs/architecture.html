Extend & Build On

career-ops is designed to be extended. Here are the main ways to plug in new tools, agents, and automations.

🔌 Add a New Mode

Create a new .md file in modes/ and add a route to SKILL.md. The mode file receives the user's input via context and has access to all User Layer files.

# modes/salary-negotiation.md ## When to invoke User says "negotiate offer" or pastes offer letter ## What to read config/profile.yml → comp targets reports/latest → score + role context ## What to output Counter-offer strategy + draft email

🤖 Build an External Agent

Any tool that can read markdown files can integrate with career-ops. The data layer is plain files — no database, no API required.

# Python agent reading career-ops data import yaml, pathlib base = pathlib.Path("C:/Users/patri/career-ops") # Read profile profile = yaml.safe_load( (base / "config/profile.yml").read_text() ) # Read tracker tracker = (base / "data/applications.md").read_text() # Read all reports reports = list((base / "reports").glob("*.md"))

📡 Key Data Interfaces

The files below are the stable "API surface" — structured data you can read/write from any external tool.

  • data/pipeline.md Add URLs here to queue for evaluation. One URL per line.
  • batch/tracker-additions/*.tsv Write TSV rows here to safely add tracker entries.
  • data/scan-history.tsv Read to check seen postings. Write to mark as seen.
  • config/profile.yml Structured YAML — read to understand candidate preferences.
  • reports/*.md Structured markdown with ## Machine Summary YAML block for downstream parsing.
  • analyze-patterns.mjs Run for JSON output on rejection patterns and score stats.
  • followup-cadence.mjs Run for JSON output on overdue follow-ups.

🔄 Automate with /schedule or /loop

Use the built-in Claude Code skills to set up recurring runs — no cron required.

# In Claude chat: # Scan every 3 days /schedule "every 3 days run /career-ops scan" # Watch pipeline until empty /loop /career-ops pipeline # Check follow-ups every Monday /schedule "every Monday /career-ops followup"

🌐 Add a New Portal to Scan

Edit portals.yml to add any company that uses Greenhouse, Ashby, or Lever. scan.mjs hits their APIs directly — zero tokens, instant results.

# portals.yml — add a company companies: - name: "The Beach Company" ats: "greenhouse" slug: "thebeachcompany" keywords: - "asset management" - "investment" - "acquisitions"

🎨 Customize the CV Design

The template is pure HTML/CSS — edit templates/cv-template.html to change fonts, colors, layout. The {{placeholders}} are replaced at generation time.

/* cv-template.html — change accent color */ .section-title { color: hsl(187, 74%, 32%); /* ← teal */ } .job-company { color: hsl(270, 70%, 45%); /* ← purple */ } /* Add a new section placeholder */ <div class="section"> <div class="section-title">{{SECTION_AWARDS}}</div> {{AWARDS}} </div>

Agents, Subagents, Skills & Hooks

The four building blocks of Claude Code automation — explained using career-ops as the live example.

🎯

Skills — Slash Commands

A markdown file that turns a phrase into a behavior. The bridge between your words and Claude's actions.

What it is

A skill is a SKILL.md file stored in ~/.claude/skills/{name}/ or .claude/skills/{name}/. When you type /career-ops in chat, Claude finds the matching SKILL.md, loads it as instructions, and executes whatever it says.

The frontmatter at the top defines how it's exposed as a slash command:

--- name: career-ops description: AI job search command center arguments: mode # the $mode variable user-invocable: true # shows up as /career-ops argument-hint: "[scan|pdf|oferta...]" --- # The rest is Claude's instructions # Written in plain markdown # $mode = whatever the user typed after /career-ops
Analogy: A skill is like a recipe card. You hand it to a chef (Claude) and say "make this." The chef reads the instructions and decides how to execute them — they're not hard-coded steps, they're guidelines a smart agent interprets.

In career-ops

There is one skill file for the whole system. It's a router — it reads the $mode argument and decides which mode file to load next.

~/.claude/skills/career-ops/SKILL.md .claude/skills/career-ops/SKILL.md

The skill itself doesn't contain the evaluation logic — it just says "if mode is oferta, read modes/oferta.md and follow it." This separation means the system logic can be updated without touching the skill definition.

Skill lookup order

You type /career-ops pdf → Claude finds SKILL.md → Reads modes/pdf.md → Executes instructions

To create your own skill

# 1. Create the folder ~/.claude/skills/your-skill-name/SKILL.md # 2. Add frontmatter + instructions --- name: your-skill-name user-invocable: true --- # Your instructions here When invoked, do X, then Y, then Z. Read $arguments for user input.
🧠

Agent — The Main Claude Session

The LLM running in your chat window. It reads context, uses tools, and executes your skill instructions.

What it is

The agent is the Claude model instance in your active session. It has a context window (everything it can "see" at once) and a toolbox (actions it can take).

Every time you send a message, the agent receives: your message + conversation history + any loaded files (CLAUDE.md, skills) + available tools. It reasons over all of this and decides what to do next.

The agent's toolbox in career-ops

ToolWhat it does
ReadReads cv.md, modes/*.md, profile.yml
WriteSaves reports, TSV files, HTML
BashRuns generate-pdf.mjs, merge-tracker.mjs
WebFetchFetches job posting pages
WebSearchComp research, company news
AgentSpawns a subagent (see next card)

How CLAUDE.md fits in

CLAUDE.md is loaded automatically every time a session starts in the career-ops folder. It's the agent's standing instructions — always in context, always active. Think of it as the agent's job description for this project.

It tells the agent: what files to read, what rules to follow, what ethical constraints apply, and how to handle the tracker. The skill files then give mode-specific instructions on top of that foundation.

Analogy: CLAUDE.md is the employee handbook. SKILL.md is the job posting that brought you here. modes/*.md are the SOPs for each task. The agent is the employee reading all three and figuring out what to do.

Context loading order

Session starts → CLAUDE.md loaded → You invoke /career-ops → SKILL.md loaded → _shared.md + mode file loaded → cv.md + profile.yml read
🔀

Subagents — Parallel Workers

Fresh Claude instances the main agent spawns to handle heavy or parallel work, without polluting the main context.

What it is

When the main agent calls the Agent() tool, it launches a subagent — a brand-new Claude instance with its own context window. The main agent writes the subagent's entire briefing as a prompt string and injects it at startup.

Subagents are used when:

  • The work is too long for the main context window
  • Multiple tasks can run in parallel (batch mode)
  • A task needs Playwright browser control without blocking the main session
  • You want clean isolation — subagent can't accidentally see unrelated context

The key constraint

Subagents start cold — they know nothing except what you inject into their prompt. This is why career-ops always injects the full text of _shared.md + the mode file into every subagent call. Otherwise the subagent wouldn't know the scoring rules.

In career-ops

Three modes spawn subagents. Here's what the actual call looks like inside modes/scan.md:

# From SKILL.md — how subagents are launched Agent( subagent_type="general-purpose", prompt=""" [full text of modes/_shared.md] [full text of modes/scan.md] Now scan portals.yml and surface any new postings not in scan-history.tsv """, description="career-ops scan" )
scan → subagent (Playwright) apply → subagent (Playwright) pipeline (3+ URLs) → subagent

⚡

Hooks — Automated Triggers

Shell commands that fire automatically at lifecycle events — before a tool runs, after Claude responds, when a file changes.

What it is

Hooks are configured in settings.json (global: ~/.claude/settings.json, project: .claude/settings.local.json). They're shell commands the harness (not Claude) executes at specific lifecycle points.

This distinction matters: hooks run even if Claude doesn't decide to do something. They're enforced by the runtime, not the model. That's what makes them reliable for automation.

Lifecycle events you can hook

EventWhen it fires
PreToolUseBefore any tool call (Read, Write, Bash…)
PostToolUseAfter a tool call completes
StopAfter Claude finishes a response
SubagentStopAfter a subagent finishes
NotificationWhen Claude sends a notification
Analogy: Hooks are like triggers in a database or Git hooks in a repo. They don't ask permission — they just fire. The model can't override them.

In career-ops

career-ops has one hook in .claude/settings.local.json — it ensures Node.js is on the PATH so scripts like generate-pdf.mjs always work:

// .claude/settings.local.json { "permissions": { "allow": [ "PowerShell($env:PATH = 'C:\\Program Files\\nodejs;' + $env:PATH; npm run doctor 2>&1)" ] } }

Practical hook patterns

// settings.json — example hooks { "hooks": { "Stop": [{ "command": "node merge-tracker.mjs" // auto-merges tracker after every response }], "PostToolUse": [{ "matcher": "Write", "command": "node verify-pipeline.mjs" // validates pipeline after every file write }] } }

How to add one

In this chat, just say: "after every response, run node merge-tracker.mjs" — the update-config skill handles writing the correct hook syntax to settings.json for you.
